# Capability Map Constitution

## Mission

Maintain a clear map of business and technical capabilities using concise, versioned specifications.

## Principles

### 1. Specs First

Important work starts with a written capability spec before implementation details are locked in.

### 2. Single Source of Intent

Each capability keeps its intent, plan, and task breakdown together under one folder in `specs/`.

### 3. Traceable Decisions

Meaningful scope or architecture changes must be reflected in the related spec or plan.

### 4. Small Increments

Capabilities should be shaped into steps that can be reviewed and delivered incrementally.

## Quality Bar

- Specs must state the capability, the user or stakeholder, and the expected outcome.
- Plans must identify constraints, dependencies, and validation steps.
- Tasks must be concrete enough to execute without reinterpreting the plan.
